아키텍처 · 데이터 전달 경로

화면 데이터가 챗봇까지 가는 4단계

페이지 안의 content script와 챗봇 화면인 iframe은 서로 다른 보안 영역이라 직접 대화할 수 없음 → 그래서 안전한 중간 단계를 거쳐 데이터를 넘김.

1
content script — 웹페이지 안에서 화면 데이터(표·차트·카드)를 추출. 단, 격리(ISOLATED) 영역이라 외부 서버 호출·패널 접근은 직접 못 함.
2
service worker (background) — 익스텐션의 중앙 관제실. 권한을 갖고 MAIN world 주입·데이터 병합·탭 관리를 수행하며, content script가 뽑은 결과를 모음.
3
side panel (얇은 셸) — 브라우저 사이드바에 떠 있는 껍데기 화면. background가 모은 데이터를 건네받아 그 안의 iframe으로 전달.
4
iframe (우리 서버 UI) — postMessage로 데이터 수신. origin 검증으로 "우리 서버에서 온 메시지"만 신뢰해 위조를 차단.
왜 굳이 거치나? ① 웹페이지 ② 익스텐션 특권 영역 ③ 우리 서버(웹 origin) — 이 세 경계를 안전하게 넘어야 하기 때문. 서로 다른 창 사이의 유일한 통로가 postMessage이고, origin 검증이 그 통로의 자물쇠 역할.
핵심 · 플러그러블 어댑터 패턴

감지 → 추출 → 공통 정규화

STEP 1

감지 (detect)

각 어댑터가 DOM 단서(클래스·태그)로 "내가 처리할 화면인가"를 스스로 판정. 결정론적이며, 맨 아래 어댑터는 항상 참이라 빈손이 없음.

→
STEP 2

추출 (extract)

판정된 어댑터가 데이터를 꺼내 공통 ScreenContext로 변환. 도메인별 화면 차이는 서버 recipe가 콘텐츠 영역(scope)만 알려주고, 구조화는 어댑터가 그 안에서 수행.

→
STEP 3

레지스트리 (우선순위 협업)

우선순위 순으로 각 어댑터가 맡은(점유한) 영역만 추출해 합침(merge). 상위가 다루지 못한 부분은 하위가 이어받아 채움.

등록된 어댑터를 우선순위 순으로 검사 — 새 프레임워크는 어댑터만 추가하면 체인에 참여

domStructure 항상 동작

표가 아니어도 div 그리드·인사카드의 반복 구조를 읽는 어댑터 최하위(구조화 폴백).

※ 위는 ISOLATED(DOM) 어댑터 체인 — 데이터레이어(MAIN world) 읽기는 별도 실행 후 병합 (프레임워크 어댑터는 현재 감지 스텁)

추출 — 어디서 뽑고, 무엇으로 라벨링

"데이터는 모델, 의미는 DOM"

전제: 라벨은 DOM에 있다 — 데이터 위치를 찾아 라벨과 매핑(대부분 어댑터 선에서 해결).

단계 (신뢰도 순)무엇비고
picked 사용자 지정챗봇 "🎯 영역 지정" → 사용자가 클릭한 영역을 최우선으로 스코프 추출보조 · UDC.run(pickedSelector)
table html 표<table> thead/tbody → 데이터 + <th> 라벨table.js
domStructure 제네릭div 그리드·인사카드: 반복구조로 데이터 영역 찾고 헤더행/라벨-값 매핑항상 동작하는 최하위 어댑터. dom-structure.js
데이터레이어 MAIN world페이지 전역(데이터모델·차트 인스턴스)을 executeScript({world:"MAIN"})로 읽음가상스크롤 전체 + 컬럼정의 라벨 + 차트
recipe 추출 레시피도메인별 콘텐츠 영역(scope include/exclude) + 접근(dataLayer)·거버넌스(mask/deny) 힌트. 구조화는 어댑터에 위임데이터 파일 recipes.jsonc · /api/recipe
text 보이는 텍스트위로 못 잡은 보이는 텍스트를 그대로 캡처 → 게시글·기사 등도 답변최후 폴백 · collectVisibleText(상한 30k)

📊 데이터(rows) = 모델

프레임워크 데이터 레이어에서 — 가상스크롤이어도 전체 보유. WebSquare 데이터모델 · React fiber memoizedProps · Vue data · 그리드 API forEachNode

🏷️ 의미(라벨/단위) = DOM

DOM 헤더(<th>·헤더 셀·aria-label) 또는 그리드 컬럼 정의(columnDefs의 headerName·단위) — 라벨이 설정에만 있을 때.

가시성 필터: 보이는 요소만 추출(숨은 SPA 탭/슬라이드 제외) · text 폴백 덕에 "추출 못함"은 진짜 빈 화면에서만.
어댑터가 둘을 합쳐 columns(label/unit)+rows로 정규화 · 병합 시 표·차트=MAIN 우선, 카드=ISOLATED · 차트는 데이터 레이어에서 charts(labels/series)로.
핵심 · 추출 레시피(recipe)

코드 말고 config로 — 도메인 적응

recipe 는 "어디를 읽고/버리고, 어디서 데이터를 읽고, 무엇을 가릴지"만 선언한다. 구조화(표/카드)는 어댑터가 한다.

🧩 recipe 가 선언하는 것

  • match — url(glob 또는 /정규식/) + domWhen → 여러 개면 priority
  • scope — include/exclude (모든 추출 계층 공통)
  • dataLayer — 전체데이터 위치 힌트(예 window.GRID)
  • mask 값 가림 · deny 추출 금지 (거버넌스)

📄 server/recipes.jsonc

{
  "name": "site-portal",
  "match": { "url": ["*/portal/site*"],
             "domWhen": ".site-portal" },
  "priority": 10,
  "scope": {
    "include": ["#contents"],
    "exclude": ["nav", ".gnb"]
  },
  "dataLayer": { "path": "window.GRID" },
  "mask": [".ssn"], "deny": [".secret"]
}
데이터 파일(recipes.jsonc)만 고치면 서버 재시작 없이 갱신 · recipe 없으면 어댑터 기본 동작(회귀 안전) · 스키마 검증 schemas.py Recipe.

질문 한 번에 일어나는 일

1
background (service worker) — 탭에 스크립트 주입·지휘, 마지막에 결과 병합. recipe 는 서버 /api/recipe(recipes.jsonc)에서 fetch.
2
base.js · UDC.run (ISOLATED) — 레지스트리·오케스트레이션: domWhen 게이트 → scope exclude/deny 선점 → include 루트 → picker → 어댑터 → 차트 → 텍스트 폴백 → mask
↳
어댑터 priority — websquare(10) → table(0) → domStructure(-10). 각자 scope 안에서 감지(detect)→추출(extract), 점유 안 된 나머지를 채움(merge).
3
readDataLayer (MAIN world) — 페이지 전역 window.GRID·Chart.instances 직접 읽기(가상스크롤 전체+차트). scope 필터 적용.
4
병합 → ScreenContext — 표·차트=MAIN 우선, 카드·텍스트=ISOLATED → 서버 /api/chat → OpenAI
프레임워크 판별(detect 단서): WebSquare $w·scwin·w2grid · React __reactFiber$ · Vue __vue_app__ — 현재 websquare 는 스텁, React/Vue 데이터 리더는 예정
