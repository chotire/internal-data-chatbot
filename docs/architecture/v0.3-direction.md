# 아키텍처 v0.3 — 익스텐션은 채널, AI 서비스가 두뇌 (방향 + PoC 데모)

> 이 문서는 **향후 방향과 이번 PoC 범위**를 정의한다. 현재 구현(as-built) 상세는 [v0.2-current-build.md](v0.2-current-build.md).
> 고객 합의(통화): 향후 아키텍처는 이 방향이 맞고, **PoC 단계 목표는 "챗봇이 화면 전용이 아니라 여러 툴을 꽂을 수 있다"를 보여주는 것**으로 충분.

---

## 0. 한 줄 요약 (쉬운말)

**익스텐션은 "화면을 읽어 깔끔하게 정리한 보고서(ScreenContext)를 만들어 보내는 사람"이고,
AI 서비스는 그 보고서를 받아 똑똑하게 답하는 두뇌다.
지금은 두뇌가 없어서 가짜 두뇌를 임시로 세워둔 것뿐이다.**

라우팅·복합질의 판단·여러 에이전트 조합은 **두뇌(AI 서비스)의 일**이고, 우리는 보고서를 잘 만들어
정해진 양식으로 넘기는 채널을 맡는다.

---

## 1. 큰 그림

```
[익스텐션 = 채널]  화면 인식·추출·거버넌스 → ScreenContext
        │  (안정적 계약: endpoint 요청/응답 양식 — 유일하게 영속)
        ▼
[AI 서비스]  LangGraph 멀티에이전트 — 라우팅·복합질의 판단·에이전트 조합·LLM 선택
        ▼
        응답 (최종답 + 출처/기여 에이전트, 스트리밍)
```

- **라우팅·멀티에이전트 조합은 AI 서비스의 몫.** 우리가 만드는 게 아니다.
- LLM이 무엇이든(OpenAI/사내/기타) AI 서비스가 흡수 → 채널은 신경 안 씀.
- 아직 AI 서비스를 못 붙여 **"가짜 두뇌"(임시 서버)로 연출**하는 단계.

---

## 2. 무엇이 남고 무엇을 버리는가

| 구성 | 역할 | 운명 |
|---|---|---|
| **익스텐션(채널)** | 화면 인식·추출·ScreenContext 생산·거버넌스 | **남는다 — 진짜 가치** |
| **schemas / recipe** | ScreenContext 계약 + 추출·마스킹 규칙 | **남는다 — 경계 정의** |
| **llm.py / 프롬프트 / 툴 / 라우팅** | AI 서비스 흉내(가짜 두뇌) | **버린다 — 임시 연출** |

→ **투자 대상은 우리 코드의 정교함이 아니라 "AI 서비스와 맞닿는 이음매(seam)의 충실도".**
이음매가 진짜와 같으면 나중에 연결 = **endpoint 주소 하나 교체**.

## 3. 지금 신경 쓸 6가지 (이음매 설계 원칙)

1. **ScreenContext 계약이 진짜 제품** — 유일하게 영속되는 인터페이스. `schemas.py`를 "내부 구조"가 아니라 **버전 있는 외부 API 계약**으로. 응답엔 최종답뿐 아니라 **출처/기여 툴/인용** 칸을 미리 만들어 둠(가짜 두뇌가 다 못 채워도 칸은 존재).
2. **버전 호환성** — 익스텐션은 업데이트 드물고 AI 서비스는 빠르게 진화. 구버전 ScreenContext도 알아듣게 **버전 필드** + backward-compat.
3. **거버넌스는 채널에서 끝낸다** — 데이터가 브라우저를 떠나는 출입문이 곧 채널. `mask`/`deny`/recipe scope는 **ScreenContext가 익스텐션을 떠나기 전**에 적용(받는 쪽이 아님).
4. **멀티스텝·스트리밍·상태 수용** — LangGraph는 여러 단계·대화 기억·중간 단계 스트리밍이 기본. 이음매를 "질문 1 → 답 1"로 못 박지 말고 thread + 스트리밍 이벤트 여지를 둠.
5. **호출 경로 결정** — 미래에 익스텐션이 AI 서비스를 직접 부를지, 얇은 우리 서버 프록시를 거칠지. 인증·egress·감사 때문에 **얇은 프록시가 한 겹 남을 가능성** 큼 → 지금 `/api/chat`을 "미래 프록시 자리"로 의식.
6. **챗봇 UI에 종속 금지** — "챗봇은 채널 하나일 뿐". ScreenContext 생산(추출 파이프라인)을 챗봇 iframe/사이드패널과 분리 → 다른 채널 재사용 가능.

## 4. PoC 데모 — 3개 툴로 "확장성" 보여주기

목표는 **확장성의 증명**(정확도·완성도 경쟁 아님). 가짜 두뇌에 **툴 레지스트리**를 두고, "툴 붙이기 = 레지스트리 항목 하나"임을 코드 구조로 보여준다.

| 툴 | 상태 | 데모 용도 |
|---|---|---|
| **screen_data** | 있음(컨텍스트로 주입) | 화면 값 즉답 — "이 현장 누적기성액?" |
| **code_interpreter**(OpenAI 네이티브) | 실동작 | 화면 데이터 분석 — "계약금액 최대 현장?" 정확 집계·추세·차트. 생성형 집계오류 보완 |
| **web_search**(OpenAI 네이티브) | 실동작 | 화면 밖 실시간 — "오늘 철근 시세?" → 출처 URL |

- **과투자 금지**: 진짜 LangGraph·진짜 RAG는 안 만든다(스텁도 이번엔 제외). 네이티브 툴 2개 + 화면데이터.
- **screen_data는 호출형 툴이 아니라 프롬프트에 항상 주입** — "툴 안 부르고 답함" = screen_data만 사용.

## 5. 라우팅 시뮬레이션 = LLM 툴-콜링 (별도 라우터 없음)

질문 문맥에 따라 screen_data만 / +code_interpreter / +web_search를 고르는 **라우팅은 LLM function-calling 그 자체로** 구현. 모델이 0개·1개·여러 개·순차로 툴을 고름 = 라우팅. (미래 LangGraph supervisor 라우팅과 같은 구조 → 충실한 예고편. 별도 분류기는 지연·오분류·비충실로 미채택.)

- **라우팅 규칙은 코드가 아니라 "툴 description + 시스템 프롬프트"에** 적는다: 화면으로 답 가능하면 툴 미호출 / 계산·집계·정렬·통계·차트→code_interpreter / 외부·실시간(시세·뉴스·환율)→web_search / 복합이면 둘 다.
- **루프**: `질문 + ScreenContext + 툴3개 → LLM → (툴 호출 요청 시 서버 실행→결과 주입→반복) → 최종답`. 최대 N라운드 가드.
- **무대 신뢰성**: 툴 description에 예시 박기 / `temperature` 낮게 / 시연 질문은 경로가 깔끔히 매핑되게 스크립트.

---

## 6. 출처 + 스트리밍 (한 루프에서 같이 나옴)

- **출처/도구 표기**: 실제 호출된 툴을 **기계적으로 추적**(모델 자가신고 의존 X) → `tools_used`로 응답에 실음. UI는 둘로 갈라 표기:
  **도구**(호출한 것 — 🧮 데이터 분석 · 🌐 웹 검색) / **출처**(답의 근거 데이터 — 📊 화면 데이터 + 🌐 웹 결과의 인용 URL).
  웹은 양쪽에 등장(행위=도구, 결과=출처). 📊는 화면이 실제 근거일 때만(§7 주석 규칙).
- **스트리밍 = 라우팅의 가시화**: OpenAI는 토큰 + 툴 호출 이벤트를 스트리밍. 사이드패널에 단계 표시.
  ```
  {type:"tool_start", tool:"web_search"}        → "🌐 철근 시세 검색 중…"
  {type:"tool_start", tool:"code_interpreter"}  → "🧮 분석 중…"
  {type:"token", text:"..."}                     → 답변 점진 출력
  {type:"final", tools_used:[...], citations:[...]} → 배지 + 출처
  ```
  이 이벤트 모양도 미래 AI 서비스의 "에이전트 단계 스트리밍"과 동일 → 또 하나의 충실한 이음매.

## 7. 테스트 예시 질문 (데모 시연 스크립트)

데모 화면은 v0.2와 동일(통합 대시보드 `server/web/demo.html`, 출처·데이터는 [v0.2 §7](v0.2-current-build.md#7-통합-데모--예시-질문-serverwebdemohtml)).
요약하면 한 화면에 ① **인사카드**(담당 PM: 박지연/토목사업부/010-9876-5432) ② **공종별 계약 요약 표**(주택 912,000·토목 975,000·플랜트 988,000·건축 900,000 백만원) ③ **현장 현황 가상스크롤 50행**(현장-01~50, MAIN `window.GRID` — DOM엔 ~8행만) ④ **차트 2종**이 섞여 있다.

아래 6개를 **배지가 점증하는 순서**(화면만 → 분석 → 웹 → 복합)로 정리했다. 표 위에서 아래로 시연하면 도구/출처 배지가 단계적으로 늘어 "화면 전용이 아니라 툴을 골라 쓴다"가 한눈에 보인다. 컬럼은 UI 표기와 동일(**도구**=호출한 것 / **출처**=답의 근거).

| # | 질문 | 도구 | 출처 | 근거 데이터 | 예상 답 / 비고 |
|---|---|---|---|---|---|
| Q1 | 담당 PM 부서와 연락처는? | — | 📊 화면 데이터 | 인사카드(domStructure section) | 토목사업부 / 010-9876-5432 |
| Q2 | 플랜트 공종의 계약 요약 금액은? | — | 📊 화면 데이터 | 공종별 계약 요약 표(table) | 988,000 백만원 |
| Q3 | 공종별 계약금액을 막대 차트로 보여줘 | — (```chart 렌더) | 📊 화면 데이터 | 공종별 요약/차트 데이터 | 앱이 막대차트로 렌더 [912000,975000,988000,900000] |
| Q4 | 현장별 기성률(누적기성액÷계약금액)을 구해 가장 낮은 5개 현장을 표로 보여줘 | 🧮 데이터 분석 | 📊 화면 데이터 | 현장 현황 50행 — **파생컬럼(기성률) 계산 + 정렬·top5** | 기성률 하위 5개 현장 표. 50행 파생지표+정렬이라 눈대중 불가 → code_interpreter 유도(강한 모델도) |
| Q5 | 오늘 국내 철근(H형강) 시세는? | 🌐 웹 검색 | `[1]` 인용 URL | 화면·내부에 없음 → 외부 실시간 | 시세 + 출처 URL. **화면 미사용이라 📊 안 켜짐** |
| Q6 | 전체 현장 계약금액의 평균과 표준편차를 구하고, 평균을 오늘 환율로 USD로 환산하면? | 🧮 데이터 분석 · 🌐 웹 검색 | 📊 화면 데이터 · `[1]` 환율 URL | 평균·표준편차=code_interpreter(50행), 환율=web_search | **복합 질의**(다출처): **표준편차는 닫힌식이 없어 모델이 본문 암산 불가 → code_interpreter 강제**, 평균 USD 환산은 web_search. (단순 "합계"는 모델이 자꾸 손으로 더해 🧮 생략+오답) |

> **배지 규칙(📊는 화면이 실제 근거일 때만 — 기계적 휴리스틱, `llm.py`):** ① 툴 미호출 → 📊(Q1~Q3) ② code_interpreter → 📊+🧮(Q4) ③ web_search 단독 → 🌐, 📊 미포함(Q5) ④ 둘 다 → 📊+🧮+🌐(Q6). 따라서 Q4의 📊+🧮는 *단일 출처(화면)+분석(도구)*이지 복합이 아니며, **복합 질의는 Q6**(화면 🧮 + 웹 🌐)만 가리킨다.

> ⚠️ **라우팅은 LLM이 결정 → "예상"이며 보장 아님**(§5). 특히 **복합(Q6)은 가장 안 미덥다** — 모델이 *암산 가능한 쪽을 건너뛴다*. "1위(max)"·"합계(sum)"는 모델이 본문에서 직접 계산해 🧮를 생략(게다가 자주 **오답**)하므로, 화면 계산 파트는 **표준편차·분산처럼 닫힌식 없는 통계**나 **파생비율+정렬**(Q4)처럼 *암산 불가*한 걸로 둬야 code_interpreter가 확실히 불린다. 단일 툴을 확실히 보여주려면 Q4(🧮)·Q5(🌐). **완전 보장**이 필요하면 행 데이터를 프롬프트에서 빼 code_interpreter에 *파일*로 주는 방식(③)으로. 강한 모델(`OPENAI_MODEL`)일수록 라우팅 안정적.

## 8. 성공 기준

데모의 목표는 **"속이 똑똑한 것"이 아니라 "겉(이음매)이 진짜와 똑같은 것".**
가짜 두뇌가 단순해도, **보고서 양식(ScreenContext) / 출처 표기 / endpoint·스트리밍 모양**이 진짜 AI 서비스와 같으면
→ 이 데모는 "축약된 가짜"가 아니라 **"진짜 구조의 예고편"**이고, AI 서비스 연결은 **주소 하나 바꾸는 일**이 된다.

> **한 문장:** 채널·양식·출입문 검열은 진짜처럼, 두뇌는 임시 가짜로 — 둘이 만나는 접점만큼은 진짜와 똑같이.
